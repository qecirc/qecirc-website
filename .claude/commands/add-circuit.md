You are helping the user add a new circuit to the QECirc library. This is a conversational workflow — gather information step by step, confirm with the user at each stage, and only proceed when they approve.

**Read `CLAUDE.md` first** to understand the data model, hierarchy, and database schema.
**Read `docs/adding-circuits-agent.md`** for a concise overview of this workflow.

---

## Interaction principle

**Ask first, research second.** Always ask the user for information before searching on your own. Only fall back to web research (e.g. Error Correction Zoo) when the user explicitly says they don't know or asks you to look it up.

---

## Phase 1: Collect inputs

Gather the following from the user. Ask for anything not yet provided. Do not proceed to Phase 2 until you have at least the required inputs.

### Required (hard stop if missing)

- **Hx matrix** — text rows, JSON, numpy-style, or file path
- **Hz matrix** — same format, must have same number of columns as Hx
- **STIM circuit** — text, file path, or stim string
- **Code distance `d`** — integer (must be provided by the user)
- **Source** — DOI, URL, or citation (refuse to proceed without provenance)

### Ask the user for (do not guess)

- **Code name** — "What is this code called? (e.g. 'Steane Code', 'Rotated Surface Code')"
- **Zoo URL** — "Do you have the Error Correction Zoo link? (e.g. https://errorcorrectionzoo.org/c/steane)"
- **Circuit name** — "What should this circuit be called? (e.g. 'Fault-Tolerant Syndrome Extraction')"
- **Tool** — "Was this generated by a specific tool? (e.g. mqt-qecc, cliffordopt, flag-at-origin)"
- **Description** — "One-sentence description? (optional)"

If the user doesn't know the code name or Zoo URL, offer to search the Error Correction Zoo using the code parameters.

## Phase 2: Inspect and validate

Before generating any files, use the helper functions to inspect the inputs and catch issues early.

### 2a. Check the code

```python
from scripts.add_circuit import check_code, find_existing_code_full

info = check_code(Hx, Hz, d=<distance>)
# Returns: {n, k, d, is_css, is_self_dual, canonical_hash}
```

Show the user: `[[n, k, d]]`, CSS status, self-dual status.

Check if the code already exists (with permutation info):

```python
match = find_existing_code_full(Hx, Hz)
# Returns ExistingCodeMatch(slug, qubit_permutation) or None
```

If found, tell the user: "This code already exists as `<slug>`. The new circuit will be added to it."

If `match.qubit_permutation` is not None, inform the user: "Your qubit ordering differs from the stored code. The circuit will be relabeled to match (permutation: `[...]`)." This is handled automatically by `add_circuit()` — validation still uses the user's original matrices (same source as the circuit).

### 2b. Inspect the circuit

```python
from scripts.add_circuit import summarize_circuit

summary = summarize_circuit(circuit_text)
# Returns: {qubit_count, gate_count, depth, crumble_url, quirk_url}
```

Show the user the metrics and the Crumble link for visual inspection.

### 2c. Extract code from circuit (optional)

If the user has a circuit but no check matrices, extract them:

```python
from scripts.add_circuit import extract_code

# For encoding circuits (first k qubits are data):
result = extract_code(circuit_text, circuit_type="encoding", k=<k>)

# For state-preparation circuits:
result = extract_code(circuit_text, circuit_type="state_prep", k=<k>)

# result.Hx, result.Hz, result.n, result.k, result.is_css
```

Encoding extraction is exact. State-prep Hz uses a heuristic for k >= 1 — prefer encoding circuits when available.

### 2d. Validate (optional, based on circuit type)

If the user says what the circuit does, run the appropriate validation:

```python
from scripts.add_circuit import validate_encoding, validate_state_prep

# For encoding circuits:
result = validate_encoding(circuit_text, Hx, Hz)

# For state-preparation circuits:
result = validate_state_prep(circuit_text, Hx, Hz)
```

If validation fails, report details and stop. Do not proceed with invalid circuits unless the user explicitly overrides.

### 2d. Dry-run preview

```python
from scripts.add_circuit import preview_circuit

result = preview_circuit(
    Hx=Hx, Hz=Hz, circuit=circuit_text,
    circuit_name="...", d=<distance>,
    code_name="...", source="...", tool="...",
)
print(result.summary())
```

Show the user what files would be generated. **Ask to confirm before proceeding.**

## Phase 3: Generate YAML files

Once the user confirms the dry run:

```python
from scripts.add_circuit import add_circuit

result = add_circuit(
    Hx=Hx, Hz=Hz, circuit=circuit_text,
    circuit_name="...", d=<distance>,
    code_name="...", source="...", tool="...",
    zoo_url="...", description="...",
)
print(result.summary())
```

Report the files written.

## Phase 4: Zoo lookup and tagging

After files are written, enrich with tags. This is the most judgment-heavy phase.

### 4a. Collect existing tags

Read all existing YAML files to build the tag vocabulary:

```bash
grep -h '^tags:' data_yaml/codes/*.yaml data_yaml/circuits/*.yaml data_yaml/tools/*.yaml
```

**Only use tags that already exist in the library.** Do not invent new tags unless the user explicitly asks.

Build the tag vocabulary dynamically by reading the YAML files — do not rely on a hardcoded list.

### 4b. Auto-tags (already applied by the pipeline)

The pipeline auto-detects and adds to the **code** YAML:
- `CSS` — if Hx·Hz^T = 0 mod 2
- `self-dual` — if rowspace(Hx) = rowspace(Hz)

These are mathematically verified. Do not remove them.

### 4c. Zoo lookup for code tags

If the code has a `zoo_url`, or if the user wants you to find one:

1. **Fetch the Zoo page**: Use `WebFetch` on `https://errorcorrectionzoo.org/c/<code-id>`
2. **Extract information**: Look for:
   - Code family (e.g. "color code", "surface code", "bicycle code", "LDPC")
   - Properties mentioned (e.g. "self-dual", "transversal gates")
   - Alternative names
3. **Map to existing tags**: Only propose tags that already exist in the library. If the Zoo mentions something not in the vocabulary, ask the user if they want to add it as a new tag.
4. **Add `zoo_url`** to the code YAML if not already present.

### 4d. Circuit tags — ask the user

Circuit tags require user input. Ask explicitly:

- **Functionality**: "Is this an encoding, state-preparation, or syndrome-extraction circuit?"
- **Fault tolerance**: "Is this circuit fault-tolerant? (ft / non-ft)" — **never guess this**
- **Special properties**: Only tag if the user or source paper explicitly claims it (e.g. `gate-optimal`, `depth-optimal`, `flag`)

### 4e. Apply tags

Edit the generated YAML files to add the `tags:` field:

- `data_yaml/codes/<slug>.yaml` — code-level tags
- `data_yaml/circuits/<code-slug>--<circuit-slug>.yaml` — circuit-level tags

### Tag rules

- **Never infer optimality** (`gate-optimal`, `depth-optimal`): only tag if the user or source paper explicitly claims it.
- **Always ask** about fault tolerance — it cannot be reliably detected from the circuit alone.
- Present suggested tags to the user for approval before writing them.
- Add `stabilizer` to code tags if not already present (all codes in this library are stabilizer codes).

---

## Phase 5: Final review

1. Read and show the final YAML files to the user (code YAML + circuit YAML).
2. Ask: "Does everything look correct? You can edit anything before we rebuild."
3. Let the user make manual corrections if needed.
4. Once approved, rebuild:

```bash
npm run db:create && npm run dev
```

5. Confirm the database rebuilt successfully.

## Error handling

| Condition                             | Action                                                    |
| ------------------------------------- | --------------------------------------------------------- |
| Source missing                        | Hard stop — ask the user to provide it                    |
| Distance missing                      | Hard stop — ask the user to provide it                    |
| Matrices have different column counts | Hard stop — report the mismatch                           |
| Circuit validation fails              | Stop — report details, suggest checking qubit ordering    |
| Non-trivial qubit permutation         | Inform user, show permutation, proceed (relabeling is automatic) |
| Tool slug not in `data_yaml/tools/`   | Ask user to confirm; note a new tool YAML may be needed   |
| Zoo URL not found                     | Continue without it — not required                        |
| Zoo page fetch fails                  | Continue — tag manually with user input                   |
