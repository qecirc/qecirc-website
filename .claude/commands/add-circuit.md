You are helping the user add a new circuit to the QECirc library. This is a conversational workflow — gather information step by step, confirm with the user at each stage, and only proceed when they approve.

**Read `CLAUDE.md` first** to understand the data model, hierarchy, and database schema.
**Read `docs/adding-circuits.md`** for the ingestion workflow and YAML format reference.

---

## Interaction principle

**Ask first, research second.** Always ask the user for information before searching on your own. Only fall back to web research (e.g. Error Correction Zoo) when the user explicitly says they don't know.

---

## Phase 1: Collect inputs

Gather the following from the user. Ask for anything not yet provided. Do not proceed to Phase 2 until you have at least the required inputs.

### Required (hard stop if missing)

- **Hx matrix** — text rows, JSON, numpy-style, or file path
- **Hz matrix** — same format, must have same number of columns as Hx
- **STIM circuit** — text, file path, or stim string
- **Source** — DOI, URL, or citation (refuse to proceed without provenance)

### Ask the user for (do not guess)

- **Code name** — "What is this code called? (e.g. 'Steane Code', 'Rotated Surface Code')"
- **Code distance** — "What is the code distance d? (leave blank if unknown — it will be computed)"
- **Zoo URL** — "Do you have the Error Correction Zoo link? (e.g. https://errorcorrectionzoo.org/c/steane)"
- **Circuit name** — "What should this circuit be called? (e.g. 'Fault-Tolerant Syndrome Extraction')"
- **Functionality** — "What does this circuit do? (encoding, state-preparation, syndrome-extraction)"
- **Tool** — "Was this generated by a specific tool? (e.g. mqt-qecc, cliffordopt, flag-at-origin)"
- **Fault tolerance** — "Is this circuit fault-tolerant? (ft / non-ft)"
- **Special properties** — "Any special properties? (e.g. gate-optimal, depth-optimal, local-connectivity)"
- **Description** — "One-sentence description? (optional)"

If the user doesn't know the code name, Zoo URL, or code family, offer to search the Error Correction Zoo (`https://errorcorrectionzoo.org/`) using the code parameters.

---

## Phase 2: Run ingestion

Once inputs are collected:

1. **Prepare input files**: Convert the user's matrices and circuit text into temporary files if needed (JSON for matrices, `.stim` for circuit).

2. **Dry run first**:

```bash
python -m scripts.add_circuit.generate \
  --hx <hx_path> --hz <hz_path> \
  --stim <stim_path> \
  --code-name "<code name>" \
  --circuit-name "<circuit name>" \
  --source "<source>" \
  --tool "<tool slug>" \
  --description "<description>" \
  --zoo-url "<zoo url>" \
  --d <distance> \
  --dry-run
```

Or use the Python API:

```python
from scripts.add_circuit import add_circuit
result = add_circuit(Hx=Hx, Hz=Hz, circuit=circuit_text, ..., dry_run=True)
print(result.summary())
```

3. **Show the summary** to the user: code status (new/existing), validation result, detected functionality, files that would be written.

4. **Check validation**: If validation failed, report details and stop. Do not proceed with invalid circuits.

5. **Ask the user to confirm** before writing files. If approved, run again without `--dry-run`.

---

## Phase 3: Tag the code and circuit

After files are written, apply tags. This is where your judgment matters.

### Check existing tags first

Read existing YAML files in `data_yaml/codes/` and `data_yaml/circuits/` to see what tags are already in use. **Prefer existing vocabulary over inventing new terms.**

### Code tags

| Category        | Examples                                                                     | Source                                             |
| --------------- | ---------------------------------------------------------------------------- | -------------------------------------------------- |
| Code type       | `CSS`, `stabilizer`                                                          | Auto-detected by scripts                           |
| Code family     | `color-code`, `surface-code`, `toric-code`, `bicycle`, `LDPC`, `Reed-Muller` | Ask user or Zoo lookup                             |
| Code properties | `self-dual`, `single-shot`, `transversal-gates`                              | `self-dual` auto-detected; others from user or Zoo |

### Circuit tags

| Category           | Examples                                                            | Source                           |
| ------------------ | ------------------------------------------------------------------- | -------------------------------- |
| Functionality      | `encoding`, `state-preparation`, `syndrome-extraction`              | Auto-detected; confirm with user |
| Fault tolerance    | `ft`, `non-ft`                                                      | **Must ask user** — do not guess |
| Special properties | `gate-optimal`, `depth-optimal`, `local-connectivity`, `flag-based` | **Only if user claims it**       |

### Tag rules

- **Never infer optimality** (`gate-optimal`, `depth-optimal`): only tag if the user or the source paper explicitly claims it.
- **Always ask** about fault tolerance — it cannot be reliably detected from the circuit alone.
- Present any suggested tags to the user for approval before applying.

### Apply tags

Edit the generated YAML files to add/update the `tags:` field in:

- `data_yaml/codes/<slug>.yaml`
- `data_yaml/circuits/<code-slug>--<circuit-slug>.yaml`

---

## Phase 4: Final review

1. Read and show the final YAML files to the user.
2. Ask: "Does everything look correct?"
3. If approved, instruct them to rebuild:

```bash
npm run db:create && npm run dev
```

---

## Error handling

| Condition                             | Action                                                    |
| ------------------------------------- | --------------------------------------------------------- |
| Source missing                        | Hard stop — ask the user to provide it                    |
| Matrices have different column counts | Hard stop — report the mismatch                           |
| Commutativity check fails             | Hard stop — matrices don't define a valid stabilizer code |
| Circuit validation fails              | Stop — report details, suggest checking qubit ordering    |
| Distance computation times out        | Continue with `d: null` — note it for the user            |
| Tool slug not in `data_yaml/tools/`   | Ask user to confirm; note a new tool YAML may be needed   |
| Zoo URL not found                     | Continue without it — not required                        |

- [ ] Source/citation recorded
- [ ] Code name matches canonical name (Zoo or literature)
- [ ] Circuit name is descriptive (not "Circuit 1")
- [ ] Validation passed (or skipped with explanation)
- [ ] Code tags: type (`CSS`/`stabilizer`) + family if known
- [ ] Circuit tags: functionality + fault-tolerance status
- [ ] No optimality tags unless explicitly claimed
- [ ] YAML files reviewed and confirmed by user
